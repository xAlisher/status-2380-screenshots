Intentional Design Changes in This Build

The following steps were deliberately removed from onboarding to reduce friction and speed up time-to-home. They are design decisions, not bugs.

  • Display name: deferred — user can set it post-onboarding from profile settings.
  • Profile picture: deferred — same post-onboarding path.
  • Seed phrase / recovery backup: removed from the onboarding critical path; expected to be prompted later (e.g., contextually when user first tries to send funds).
  • Biometric auth: not offered in this build flow.

UX consideration: deferring seed phrase backup increases conversion but raises the question of when and how prominently it is surfaced later. This report flags the first-session experience; follow-up review should verify the deferred backup prompt exists and is hard to dismiss.

01 — Welcome

TitleWelcome to Status
Subtitle"The open-source, decentralised wallet and messenger"
Primary actionTap Create profile
SecondaryLog in (existing profile)
Other elementsAuto-playing carousel (3 slides), language selector (EN), "Third-party services enabled" link, ToS / Privacy Policy links
Findings ✓ One clear primary CTA, visually differentiated from secondary
✓ Plain language, no crypto jargon on entry screen
⚠ "Third-party services enabled" is 9px grey text — easy to miss; users unaware of data sharing default
⚠ Carousel auto-plays with no pause — WCAG 2.1 SC 2.2.2 violation for users with vestibular disorders
✗ No progress indicator (acceptable at entry, but no onboarding map shown anywhere in subsequent screens either)
Suggestions Surface "Third-party services" choice prominently (toggle or highlighted link).
Add autoPlay: false default or a pause button to the carousel.
Show a 4-dot step indicator starting on the next screen so users know how long onboarding is.

02 — Start Fresh

TitleCreate profile — "How would you like to start using Status?"
Primary actionTap Let's go! (Start fresh)
SecondaryUse a recovery phrase · Use an empty Keycard
Findings ✓ Clear visual hierarchy: new-user path prominent, import paths de-emphasised
✓ Illustrative icon reinforces "fresh start" concept
✗ No step indicator (still no X of Y)
⚠ "Use an empty Keycard" shown to all users — niche hardware wallet flow adds cognitive noise for new users who don't own a Keycard
✗ No rationale for why there are three paths; new users may be confused by the choice
Suggestions Move Keycard option behind an "Advanced" disclosure.
Add inline tooltip: "Already have a Status account? Import your recovery phrase."
Add step indicator (e.g., Step 1 of 3).

03 — Create Password

TitleCreate profile password — "This password can't be recovered"
FieldsChoose password (min 10 chars) · Repeat password
Strength meterInline bar + checklist (Lower case / Upper case / Numbers / Symbols). States observed: Very weak → Okay → Good → Very strong
Primary actionConfirm password (enabled only when both fields match)
Findings ✓ Inline requirements visible without submit attempt
✓ "Passwords match" confirmation shown in real time
✓ Reveal-password toggle on both fields
✗ "This password can't be recovered" — alarming framing with no constructive guidance (no mention of seed phrase as the real recovery path)
✗ No step indicator
⚠ On-screen keyboard language (Polski) doesn't match app UI language (EN) — keyboard locale follows OS, not app locale; characters may be remapped unexpectedly
⚠ Confirm button enables at "Good" strength (all classes met but no symbols) — no minimum strength gate beyond character class requirements
✗ adb input text injects text at the Android IME layer but Qt QML fields accumulate injected text without honouring CTRL+A / DEL for field clearing — indicates Qt–Android input bridge inconsistency (testability bug)
Suggestions Replace warning with constructive framing: "Your seed phrase is your only backup — you'll set it up next."
Require minimum "Good" strength before enabling Confirm.
Fix keyboard locale mismatch: sync IME language to app's active locale.
Screens not shown in this build's flow: Display name, Profile picture, Seed phrase intro, Seed phrase display, Seed phrase verification, Biometric authentication. These were expected per the plan but did not appear in build 2.38.0-rc.1-61. Whether they exist behind a later prompt (e.g., after first login) is not verified in this session.

04 — Enable Push Notifications

ContextBottom sheet modal over wallet home, appears immediately after password confirmation
Primary actionContinue (→ OS permission dialog, not observed)
SecondaryMaybe later
Body copy"Notifications include alerts, sounds, and icon badges… Status uses a device-local service to deliver notifications, ensuring they remain private and do not pass through any third-party or centralized servers."
Findings ✓ Privacy rationale provided before requesting permission — best practice
✓ "Maybe later" option respects user autonomy
✗ "Internet connection lost. Reconnect to ensure everythi…" error banner persists across all subsequent screens — alarming first impression for brand-new user; no retry button visible in truncated banner
✗ OS-level push permission dialog never appeared after tapping Continue — possible that the OS permission was already granted from the previously-installed nightly, or a bug prevents the dialog from showing on fresh install
✗ Timing: modal appears before user has set display name or seen any content — the app has no identity context for "why notifications?" yet
Suggestions Move notification request to after first meaningful action (e.g., first message received or first chat joined).
Show full error banner text; add a "Retry" button.
Verify OS permission dialog is triggered correctly on clean installs.

05 — Tutorial Tip ("To open app menu")

TitleTo open app menu
ContentAnimated arrow gesture illustration showing swipe-from-left for sidebar navigation
Primary actionContinue
SecondaryMaybe later (bottom-left)
Findings ✓ Single-gesture tutorial is focused and quick
⚠ Placed immediately after notifications modal — two consecutive modals increase dismissal fatigue
✗ No indication of how many tutorial tips remain (is this 1 of 1 or 1 of 5?)
✗ Gesture illustration is small and the arrow animation speed may be too fast to register for new users
⚠ The sidebar it describes contains Communities, DMs, Settings, Profile, Notifications — but none of these are contextualised for the user yet
Suggestions Combine with an interactive highlight overlay rather than a modal.
Show this tip contextually on first time the user tries to navigate, not as a standalone modal.
If multiple tips exist, add a dot indicator.

06 — Home (All Accounts / Wallet)

TitleAll accounts
ContentMulti-chain portfolio view (ETH, MATIC, AVAX, OP). Tabs: Assets · Collectibles · History. CTAs: Ways to buy · Receive. Bottom nav: Messages · Wallet · Activity.
Landing contextFirst screen a new user sees after onboarding — no profile name, no seed phrase backup, no contacts
Findings ✗ "Internet connection lost" error persists — network error on first home view is a poor first impression
✗ User has no display name or avatar set — the "DP" placeholder in the top-right suggests an auto-generated identity, but this is not explained
✗ No post-onboarding prompt to back up seed phrase — wallet is active but unprotected
⚠ Wallet is the default landing tab, not Messages — positioning Status as "wallet first, messenger second" may conflict with user mental model if they signed up for private messaging
✓ "Ways to buy" and "Receive" are immediately actionable for users ready to fund
✓ Multi-chain support visible upfront
Suggestions Display a "Finish setup" banner (backup seed phrase, set display name) until dismissed.
Show a welcome empty-state message: "You're in! Add a friend or back up your wallet to get started."
Consider defaulting landing to Messages tab for new users (matches product positioning as a messenger).
